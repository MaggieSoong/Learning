Foundations
Supervised vs Unsupervised vs Reinforcement Learning
Core+

Supervised: You have labelled data (X → y). Model learns a mapping. Examples: regression, classification.

Unsupervised: No labels. Find structure in data. Examples: k-means clustering, PCA, autoencoders. Used for customer segmentation, anomaly detection.

Reinforcement: Agent takes actions, receives rewards. Less common in most consulting work but relevant for pricing optimisation and decision-making systems.

💬 Likely Interview Q"Walk me through a supervised learning problem you'd set up to predict customer churn."
Bias-Variance Tradeoff
Core+

Bias: Error from wrong assumptions. High bias = underfitting (model too simple, misses signal).

Variance: Error from sensitivity to fluctuations in training data. High variance = overfitting (model memorises noise).

Total Error = Bias² + Variance + Irreducible Noise. You want to find the sweet spot.

# Signs of overfitting: Train accuracy: 99% | Val accuracy: 72% ← big gap = overfit # Fixes: regularization (L1/L2), dropout, more data, simpler model
💡 TipIn credit and risk models, regulators care deeply about overfitting. Always mention validation on out-of-time samples.
Key Algorithms: When to Use What
IMP+
  • Linear/Logistic Regression — interpretable, fast, great baseline; used in regulated industries for interpretability
  • Decision Trees — super interpretable, prone to overfitting; use for early exploration or with ensembles
  • Random Forests — reduces overfitting vs single tree, good all-rounder; often the go-to for prediction tasks on tabular data
  • Gradient Boosting (XGBoost, LightGBM) — state-of-the-art for tabular data; slower to train but often wins competitions
  • SVMs — great for high-dimensional data; can be slow to train on large datasets
  • Neural Networks — powerful but black-box; better for images, text, sequential data
💬 Interview Q"You have a tabular dataset with 100K rows. Which algorithm do you pick first, and why?"
Cross-Validation & Proper Test Strategy
Core+

Why: You need to estimate how your model performs on unseen data. Training accuracy lies.

k-Fold CV: Split data into k folds. Train on k-1, test on 1. Repeat k times. Average the scores. Reduces variance in your estimate.

Time-Series CV: Never shuffle! Use expanding window: train on [t1, t2), test on [t2, t3). Models must predict the future, not the past.

Stratified CV: For imbalanced data, ensure each fold has the same class distribution.

💡 TipCommon mistake: CV on full data, then test on held-out set. The held-out set should have been removed before CV.
Evaluation Metrics: Accuracy, Precision, Recall, F1, AUC-ROC
Core+
  • Accuracy: (TP+TN)/(TP+TN+FP+FN). Good baseline, misleading for imbalanced data.
  • Precision: TP/(TP+FP). When you predict positive, how often are you right? Minimize false alarms.
  • Recall: TP/(TP+FN). Of all true positives, how many did you catch? Minimize missed cases.
  • F1: Harmonic mean of precision & recall. Good single metric when both matter.
  • AUC-ROC: Plots TPR vs FPR at different thresholds. Good for imbalanced data. AUC = probability the model ranks a random positive above a random negative.
💬 Interview Q"You're predicting loan defaults (1% class). Accuracy of 99% is great, right?" No — this model always predicts "no default." Use AUC-ROC or F1 instead.
Hyperparameter Tuning & Regularization
IMP+

Hyperparameters: Learning rate, tree depth, lambda (regularization), dropout rate, etc. Set before training.

Tuning methods: Grid search (exhaustive), random search, Bayesian optimisation (smarter).

Regularization (combat overfitting):

  • L1 (Lasso): Penalizes |w|. Encourages sparsity (some weights → 0). Good for feature selection.
  • L2 (Ridge): Penalizes w². Shrinks all weights uniformly. Generally more stable.
  • Early Stopping: Monitor validation loss. Stop training when it starts increasing.
  • Dropout: (Neural nets) Randomly turn off neurons during training. Reduces co-adaptation.
Ensemble Methods Deep Dive
IMP+

Bagging (Bootstrap Aggregating): Train independent models on different random subsets of data. Average predictions. Reduces variance. Example: Random Forests.

Boosting: Train models sequentially. Each new model focuses on examples the previous ones got wrong. Reduces bias. Examples: AdaBoost, Gradient Boosting, XGBoost.

Stacking: Train multiple diverse models (base learners), then feed their predictions into a meta-learner. Can combine different model types.

When to use which: Bagging → low bias high variance (e.g., deep trees). Boosting → high bias (e.g., shallow trees, weak learners). Stacking → squeeze every % of performance.

💡 TipEnsembles work best when base models are diverse (different algorithms, different feature subsets) and uncorrelated.
Model Interpretability & Explainability
Core+

Why it matters: In regulated industries (finance, healthcare), stakeholders need to understand *why* the model made a decision.

  • Feature Importance: Which features matter most? Tree-based models give built-in importance (Gini, gain). Others use permutation importance.
  • SHAP Values: Game-theoretic approach. Tells you each feature's contribution to each prediction. Local + global explanations.
  • LIME: Approximate model locally with interpretable model (e.g., linear). Explains individual predictions.
  • Partial Dependence Plots: Marginalise out other features. See how model output changes with one feature.
  • Feature Effect vs Feature Importance: Importance = does it matter? Effect = in what direction & magnitude does it push predictions?
💬 Interview Q"Your black-box model is 5% more accurate than logistic regression. But the regulator won't approve it. What do you do?" Answer: Use SHAP/LIME to explain it, or stick with logistic regression.
MLOps & Model Deployment
IMP+

Model Serving: How do you get predictions from a trained model in production? REST API (FastAPI, Flask), batch scoring, streaming.

Monitoring & Drift Detection: In production, data changes. Features shift (covariate shift), labels shift (concept drift). Monitor with PSI (Population Stability Index), track prediction distribution.

A/B Testing: Deploy new model to 10% of users. Compare metrics vs baseline. Statistical significance?

CI/CD for ML: Automate retraining, testing, deployment. Tools: MLflow, Kubeflow, DVC, Jenkins.

# Basic drift detection pattern: if PSI(current_data, historical_data) > threshold: alert("Data drift detected. Retrain model.")

Core Concepts
What is Machine Learning?
Core+

Traditional programming: you write explicit rules (if X > 5, do Y). Machine Learning: you show the model examples, it learns the pattern.

The ML Pipeline: Raw data → Cleaning → Feature engineering → Model training → Evaluation → Deployment → Monitoring.

# Traditional if income > 50k and credit_score > 700: approve_loan() # ML approach model = train_on_historical_approvals_and_defaults() prediction = model.predict([income, credit_score, age, ...])
Data: Features, Labels, Training vs Test
Core+

Features (X): The inputs. Example: [age, income, credit_score]. Can be numerical or categorical.

Labels (y): The target you're predicting. Example: [approved, denied]. Only available in training data.

Training set: Data you use to train the model. Model "sees" labels here.

Test set: Held out data. Model has never seen these labels. Used to measure performance on unseen data.

Golden rule: Never train on test data. It causes overfitting and lies about performance.

The Curse of Dimensionality
IMP+

More features = more data needed to train well. In high dimensions, most data points are far apart → model struggles to generalise.

Example: 100 features, 1000 rows. Each feature has ~10 unique values per row on average. Sparse! Model overfits.

Solutions: Feature selection (keep only important ones), dimensionality reduction (PCA), regularization, or collect more data.

💡 TipRule of thumb: aim for at least 10-20 training examples per feature. 100 features with 1000 rows? You're cutting it close.
Imbalanced Classes & Sampling
Core+

Real world: positive class (e.g., fraud, default, churn) is rare. 99% negative, 1% positive. Standard models ignore the minority class.

  • Over-sampling: Duplicate minority examples. Fast but can overfit.
  • Under-sampling: Remove majority examples. Loses data, may miss patterns.
  • SMOTE: Synthetic Minority Over-sampling Technique. Generate synthetic minority examples by interpolating between real ones.
  • Class weights: Tell the model: "I care more about getting the minority class right." Penalize minority misclassifications more.
Feature Engineering & Scaling
Core+

Feature Engineering: Create new features from raw data. Example: from [birth_date], create [age, is_over_65]. Domain knowledge + experimentation.

Scaling: Models like KNN, SVM, neural nets are sensitive to feature magnitude. Standardize (mean 0, std 1) or normalise (min-max to [0,1]).

Encoding Categorical Data: One-hot encoding (create binary column per category), ordinal encoding (assign numbers), embeddings (learned representations).

# Example: one-hot encoding color = ['red', 'blue', 'red'] → color_red = [1, 0, 1] → color_blue = [0, 1, 0]
Loss Functions & Optimization
IMP+

Loss function: Measures how wrong the model is. Lower loss = better model. Examples: MSE (regression), cross-entropy (classification).

Optimization: Adjust model weights to minimize loss. Gradient descent: compute gradient (slope of loss), step in the direction that reduces it.

Learning rate: Step size. Too high → overshoot minima (diverge). Too low → crawl slowly (take forever).

Variants: SGD (update per sample), mini-batch (update per batch), Adam (adaptive learning rate per parameter).

💡 TipAdam is often the "set it and forget it" choice. If loss doesn't decrease, try lowering learning rate or checking for bugs in data preprocessing.

Transformers & Attention Mechanism
Core+

Transformers: Architecture that powers modern LLMs (ChatGPT, Claude, Llama). Replaced RNNs because they parallelize better and handle long-range dependencies.

Attention: Mechanism that lets the model focus on relevant parts of the input. Multi-head attention = multiple "views" of relevance.

Positional Encoding: Transformers don't have inherent sequence order (unlike RNNs). Positional embeddings tell the model word order.

Self-Attention: Each token attends to all other tokens. Learns what to focus on. Queries, Keys, Values (Q, K, V) framework.

# Attention equation (simplified) Attention(Q, K, V) = softmax(Q * K^T / sqrt(d_k)) * V # High scores = strong attention
Pre-training, Tokenization & Embeddings
Core+

Tokenization: Split text into tokens (words, subwords, or bytes). LLMs use subword tokenization (BPE, SentencePiece) → better for rare words.

Embeddings: Each token maps to a dense vector (e.g., 768-dim). "King" - "Man" + "Woman" ≈ "Queen" (vector arithmetic captures semantics).

Pre-training: Train on huge corpus (Common Crawl, Books, Wikipedia) with self-supervised objectives:

  • MLM (Masked Language Model): Hide 15% of tokens, predict them. BERT uses this.
  • Causal LM (Next Token Prediction): Given context, predict next token. GPT uses this.

Result: Model learns grammar, facts, reasoning without labeled data.

Fine-tuning vs RAG vs Prompt Engineering
Core+

Decision Framework: What's the best way to adapt a pre-trained LLM for your task?

  • Prompt Engineering: Craft the prompt carefully. In-context learning. Fast, cheap, no training. Good for general tasks. Limited by context window.
  • RAG (Retrieval-Augmented Generation): Embed company documents, retrieve relevant ones at inference time, pass to LLM. Good for private/domain data. No model updates needed.
  • Fine-tuning: Update model weights on domain data. Expensive (compute + data labeling). Best performance on specialized tasks. Slower inference sometimes.
💡 TipWhen to use which: Task straightforward? Prompt engineering. Have domain docs? RAG. Accuracy critical & have $$? Fine-tune.
Prompt Engineering & Few-Shot Learning
IMP+

Prompt Design: How you ask the LLM matters. Be specific. Chain of thought (CoT) = ask the model to explain its reasoning step-by-step before answering.

Few-Shot Learning: Give 1-5 examples in the prompt. Model learns the pattern and applies it to new inputs.

# Zero-shot (bare minimum) "Classify this review: 'Great product!'" # Few-shot (better) "Examples: 'Love it' = positive, 'Broke in 2 days' = negative. Now classify: 'Great product!'"
💬 Interview Q"You need to classify 10K support tickets. No budget for fine-tuning. What do you do?" Few-shot prompting + batch API.
Agents & Tool Use
IMP+

LLM Agents: Instead of just generating text, the LLM can take actions. Think, decide which tool to use, use it, get results, repeat.

Tool Use / Function Calling: Define functions (APIs) the agent can call: fetch_weather(), search_web(), query_database(). Agent chooses which to invoke.

ReAct Pattern: Reasoning + Acting. Agent reasons about what to do, takes action, observes result, refines reasoning.

  • Q: "What's the weather in Paris? How does it compare to London?"
  • Agent: "I need to fetch weather for both cities." → call fetch_weather("Paris") → call fetch_weather("London") → reason & answer.
💡 TipAgents can solve multi-step problems that pure prompt engineering can't. But they're slower & less reliable. Use for complex reasoning tasks, not simple classification.
LLM Evaluation & Benchmarks
IMP+

Automatic Metrics: (Fast but imperfect)

  • Perplexity: How surprised is the model at the ground truth? Lower = better. Used for language modeling.
  • BLEU, ROUGE: N-gram overlap with reference. Good for summarisation/translation. Not perfect (multiple good answers exist).

Benchmarks:

  • MMLU: 57K multiple choice questions (science, history, law, ...). Tests general knowledge.
  • HumanEval: Code generation. Does the model write correct Python/etc?
  • Domain-specific: FinBENCH (finance), BioBENCH (biology), etc.

Human Evaluation: Gold standard but expensive. Rating scale (1-5), pairwise comparisons, red-teaming.

Applications
Hallucinations & Limitations
IMP+

Hallucination: Model invents facts that aren't true. "What's the capital of Atlantis?" → LLM might confidently give a wrong answer.

Why: LLMs predict the next token based on patterns, not understanding. They don't check facts against a database.

Mitigations:

  • Use RAG: retrieve facts from a source of truth first
  • Fine-tune on accurate data
  • Use smaller models that are more conservative
  • Add guardrails: "If you don't know, say 'I don't know'"
  • Fact-check outputs programmatically when possible
💡 TipIn regulated/high-stakes domains (finance, healthcare), assume hallucination is possible. Always validate LLM outputs.
AI Readiness Assessment
Nice+

When evaluating whether an organisation is ready to deploy AI/LLM solutions, consider:

  • Data maturity: Do they have clean, labeled, sufficient data? Can it be accessed reliably?
  • Infrastructure: Can they run/serve models? Do they have ML ops tooling?
  • Governance: Are there policies for model validation, audit trails, responsible AI?
  • Talent: Do they have ML engineers, data scientists? Or need external help?
  • Business case: What's the ROI? Is it solving a real problem or a solution in search of a problem?
  • Change management: Are employees ready? Will they use it? Any resistance?

Word Embeddings (Word2Vec, GloVe, FastText)
Core+

Word2Vec: Predict context from word (skip-gram) or word from context (CBOW). Learns dense vectors. "King" - "man" + "woman" ≈ "Queen".

GloVe: Combines count-based (co-occurrence matrices) and predictive (Word2Vec) methods. Often slightly better than Word2Vec.

FastText: Like Word2Vec but operates on character n-grams. Better for rare words & morphology. Good for non-English languages.

Why embeddings? Raw text is high-dimensional and sparse. Embeddings are dense, low-dimensional, and capture semantics.

💬 Interview Q"Why not just use one-hot encoding for words?" It's sparse (100K vocab = 100K features), doesn't capture similarity. Embeddings solve this.
Recurrent Neural Networks (RNNs, LSTMs, GRUs)
Core+

RNNs: Process sequences one step at a time. Hidden state at step t depends on input & hidden state from t-1. Captures temporal/sequential info.

Problem: Vanishing gradient. As sequences get long, gradients shrink → hard to learn long-range dependencies.

LSTMs (Long Short-Term Memory): Gating mechanism (input, forget, output gates) controls information flow. Solves vanishing gradient. Still used for some tasks.

GRUs: Simpler than LSTMs (fewer gates). Similar performance, faster to train.

Today: Transformers replaced RNNs for most tasks (faster, better, scales better). But RNNs are still useful for some applications.

Text Classification & Sentiment Analysis
Core+

Task: Assign labels to text. Examples: spam vs not spam, positive/negative/neutral sentiment, topic classification.

Approaches:

  • Bag-of-words → logistic regression or SVM. Fast baseline.
  • TF-IDF features → any classifier. Good for interpretability.
  • Word embeddings → feed to neural net. Better semantics.
  • Pre-trained transformers (BERT, DistilBERT) → fine-tune. SOTA, but slower & requires GPU.
# Sentiment example text = "This movie was amazing!" embedding = word2vec(text) # average word vectors sentiment = classifier.predict(embedding) # positive
Named Entity Recognition (NER) & Relation Extraction
IMP+

NER: Identify entities (people, locations, organisations) and their types in text. Example: "Apple CEO Tim Cook" → [PERSON: Tim Cook, ORG: Apple].

Approaches: Rule-based (regex, dictionary lookup), BiLSTM-CRF (sequence tagging), transformer-based (BERT + linear layer).

Relation Extraction: Find relationships between entities. Example: "Tim Cook works at Apple" → relation: [PERSON, works_at, ORG].

Use cases: Knowledge graphs, information extraction from documents, financial news analysis, contract parsing.

💡 TipFor low-resource domains (legal documents, medical texts), fine-tune a pre-trained BERT on domain-specific data.
Question Answering & Reading Comprehension
Nice+

Task: Given a context passage and a question, find or generate the answer.

Extractive QA: Answer is a span in the context. Model learns to predict start & end positions. Example: SQuAD benchmark. BERT fine-tuned for QA works well.

Generative QA: Answer is generated (not from context). More flexible, harder. Examples: T5, GPT models.

Open-domain QA: No context given. First retrieve relevant documents (BM25, dense retrieval), then extract/generate answer.

💬 Interview Q"Build a QA system for customer support docs with 10K pages." Combine BM25 retrieval + BERT extractive QA, or use LLM with RAG.
Machine Translation
Nice+

Task: Translate text from one language to another.

Seq2Seq (Sequence-to-Sequence): Encoder-decoder. Encoder compresses source text into a context vector. Decoder generates target text step-by-step.

Attention: Decoder attends to relevant encoder outputs at each step (not just final vector). Big improvement.

Transformers: No recurrence, fully parallel. GPT/T5 style. SOTA.

Challenges: Rare words, idioms, long-range dependencies, domain mismatch. Back-translation for data augmentation.

Sequence-to-Sequence & Summarisation
Nice+

Seq2Seq Architecture: Encoder-decoder framework. Encoder processes input sequence, decoder generates output sequence. Used for translation, summarisation, Q&A.

Attention Mechanism: At each decoder step, attend to relevant encoder hidden states. Without it, decoder only sees a single compressed vector (information bottleneck).

Beam Search: Decoder generates tokens one-by-one. At each step, keep top-k hypotheses. Avoids greedy locally-optimal choices.

Abstractive vs Extractive Summarisation:

  • Extractive: Select important sentences from source. Fast, faithful to source, but may be incoherent.
  • Abstractive: Generate new sentences (rewrite). More human-like. Harder — requires seq2seq. May hallucinate.
